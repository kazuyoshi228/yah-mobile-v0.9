# `chat.yah.mobi` 開発・運用向け Firestore連携・プレフィックス指示書

このドキュメントは、`chat.yah.mobi`（チャットシステム）が販売サイト（`yah.mobile`）と同一のFirebaseプロジェクトを共有し、データベース（Firestore）を安全かつ効果的に連携するためのアーキテクチャ・ルールを定義したものです。

**必ずこのルールに従って開発を進めてください。**

---

## 1. 大原則：プレフィックス（`chat_`）ルールの徹底

販売サイトのデータとチャットシステム（AIエージェント等）のデータが混ざってカオス化するのを防ぐため、**チャットシステムが作成するすべてのルートコレクションには必ず `chat_` のプレフィックス（接頭辞）を付けてください。**

### 許可されるコレクション名の例（チャットシステム側）
- `chat_sessions`（チャットのセッションやスレッド）
- `chat_messages`（チャットのメッセージ履歴）
- `chat_prompts`（AIエージェントのプロンプト管理）
- `chat_agent_logs`（エージェントの動作ログ）

### 禁止事項
- `sessions`, `messages`, `logs` といった、将来的に販売サイト側でも使いそうな一般的な名前をプレフィックスなしでルートコレクションとして作成してはいけません。

---

## 2. ユーザー認証と `users` コレクションの共有

Firebase Authentication は両サイトで完全に共有されます（SSO: シングルサインオン）。
販売サイト側ですでに存在する `users` コレクションを、チャット側でも共有して使用します。

### ルール
- **ユーザーの基本情報（名前、メアド等）**: 販売サイト側で管理される `users/{userId}` を参照してください。
- **ユーザー個別のチャットデータ**: ルートの `chat_sessions` 等に保存して `userId` フィールドで紐づけるか、`users/{userId}/chat_history` のようにサブコレクションとしてぶら下げる設計にしてください（サブコレクション名にも `chat_` を含めることを推奨します）。
- **ユーザー情報の更新制限**: チャットシステム側から `users` ドキュメントの販売関連フィールド（例: stripeCustomerId など）を不用意に上書きしないように注意してください。

---

## 3. 販売サイトデータへのアクセス（AI連携のメリット）

データベースを共通化する最大のメリットは、チャットAIがユーザーの**「購入履歴」や「プラン情報」**を即座に参照できることです。

- **参照可能なコレクション**: `orders`（購入履歴）, `plans`（販売中のプラン一覧）, `esim_links`（発行済みeSIM情報）
- **利用例**: AIがユーザーの問い合わせに答える際、裏側で `orders` コレクションを `userId` で検索し、「昨日ご購入いただいた〇〇プランですね」といったパーソナライズされた対応を行います。
- **注意点**: チャットシステム側から販売サイトのデータ（`orders` など）は **「読み取り専用（Read-Only）」** とし、絶対に更新・削除を行わないでください。

---

## 4. セキュリティルールの管理

Firestoreのセキュリティルール（`firestore.rules`）は1つのFirebaseプロジェクトにつき1つしか存在しません。

- チャットシステム用のルールを追加する場合は、既存の販売サイト用のルールを壊さないように、明確にセクションを分けて追記してください。

**追加するルールのイメージ:**
```javascript
// ─── Chat System Collections (chat_*) ───────────────────────────────
match /chat_sessions/{sessionId} {
  allow read, write: if isAuthenticated() && request.auth.uid == resource.data.userId;
}

match /chat_messages/{messageId} {
  allow read, write: if isAuthenticated() && request.auth.uid == resource.data.userId;
}
```

---

## 5. デプロイ時の注意（ホスティングとデータベースの違い）

Firebaseの仕様上、「データベースのあるプロジェクト」と「Webサイトのホスティング先」を明確に区別してデプロイする必要があります。

- **フロントエンドのホスティング（Webサイト）**: 
  チャットシステムのフロントエンドは、Firebase Hostingの別サイトとしてデプロイします。
  `firebase.json` 等でターゲットを明記し、必ず **`site: chat-yah-mobi-v2`** に対してデプロイを行ってください。販売サイトのHosting領域を誤って上書きしないよう厳重に注意してください。

- **Firestoreルール等のデプロイ（データベース）**: 
  データベース（Firestore）自体は販売サイトと同じプロジェクト（`yah-mobile-v1-3ed24`）を共有しています。
  そのため、もし `chat.yah.mobi` 側からFirestoreのセキュリティルール（`firestore.rules`）をデプロイする場合、販売サイト側のルールを古いファイルで上書き（デグレ）してしまわないよう、ルールの管理・デプロイ権限には十分に注意してください。
