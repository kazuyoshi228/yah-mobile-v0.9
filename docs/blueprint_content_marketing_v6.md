## Executive Summary

**「グルメで信頼を作り、通信で回収する」**

magazine.yah.mobi は、日本在住チームにしか書けない**グルメ一次情報**（実訪問・実価格・実写真・地図）で訪日旅行者とAI検索エンジンの両方から信頼を獲得し、少数精鋭の**マネーページ（eSIM比較・購入導線）8本**で回収するコンテンツファネルである。

- **グルメ記事 = 集客とAI引用の資産**（月10本の6割を投下）。AIは一次情報を引用せざるを得ない。
- **マネーページ = 回収装置**（8本×4言語だけを最高品質で維持）。全グルメ記事から通信ブリッジで内部リンク。
- **福岡 = 旗艦クラスタ**。在住地の情報深度で「誰にも負けない都市」を1つ作り、yah.homes クロスセルも最大化。その後、東京・大阪へ展開。

財務上の位置づけ（42ヶ月シート・magazineシナリオ Base）: 単体では**単月黒字化 M19・累計回収 M29・42ヶ月合計 +約720万円**。鯨BD（早い・大きい）との統合シナリオでは**累計回収 M13**。magazine は「遅いが複利で効き、止めても資産が残る」長期エンジンであり、単体の速効性を求めない。

## 1. 環境分析：AI引用時代の勝ち筋

### 1-1. AI検索がコンテンツに与える影響（2026年時点の実勢）

| コンテンツ類型 | AI検索の影響 | v6の扱い |
|---|---|---|
| 設定・HowTo（eSIM設定手順等） | **最悪**。AIが回答内で完結、ゼロクリック化。 | 主軸から降格。マネーページの一部として維持のみ |
| 一般情報（eSIMとは等） | 悪い。AI回答で完結 | 作らない |
| 比較・購買直前（おすすめ・比較・料金） | 中程度。AI回答に引用+購入はサイトで発生 | **マネーページ8本に集約** |
| **一次ローカル情報（グルメ・現地の今）** | **良い**。AIは生成できず引用元を必要とする。鮮度が武器 | **主軸。月産の6割** |
| ブランド指名（yah.mobile とは） | 影響なし | llms.txt・比較データで担保（実装済み） |

### 1-2. yah. が既に持っているAI引用アセット

- **yah.mobi/llms.txt（実装済み）**: 実勢プラン価格・Airalo との価格差を動的生成してAIクローラーに提供。「日本のeSIMで最安は？」への回答素材を自ら供給している。
- **magazine の器（実装済み）**: 4言語・カテゴリ・Schema対応構造・購入CTA。
- 今後追加: magazine.yah.mobi/llms.txt（記事一覧の自動反映）、FAQPage/HowTo/Article Schema の全記事付与（v5 §4の5条件は引き続き有効）。

### 1-3. 競合の空白（v5分析の継承＋グルメ視点の追加）

v5の分析（韓国語・繁体字のeSIM情報の質が低い）は正しいが、グルメ視点ではさらに大きな空白がある：**競合eSIMメディア（Airalo/Holafly/Saily のブログ）は日本のグルメ一次情報を構造的に書けない**（日本在住チームがいない）。韓国・台湾の旅行ブログは情報が古く、通信への導線を持たない。「実訪問した最新のグルメ情報 × その場で必要な通信」を一体で書けるのは現状 yah. だけである。

## 2. ペルソナ（v5の3人をグルメ起点に再定義）

**ジヨン（韓国・26歳・会社員）**: 友人3人と初の福岡週末旅。Instagram と Naver で「후쿠오카 맛집（福岡グルメ）」を検索。屋台ともつ鍋の行き方・並び方・予算を知りたい。→ 福岡グルメ記事（韓国語）→ 通信ブリッジ「行列状況はGoogle Mapsでリアルタイム確認。データは大丈夫？」→ eSIM購入。

**シャオメイ（台湾・32歳・夫婦旅行)**: 大阪・京都・福岡10日間。PTT/Dcard と ChatGPT で「日本 美食 行程」を調査。福岡では一棟貸しに泊まりたい。→ 福岡グルメ記事（繁体字）→ yah.homes クロスセル ＋ 10日間データ容量のマネーページ → 両方購入。

**マーカス（アメリカ・28歳・食べ歩きバックパッカー）**: 東京・大阪・福岡3週間。Perplexity と Reddit で「best ramen fukuoka locals」を検索。→ 英語グルメ記事（AI引用経由）→ 大容量プランのマネーページ → eSIM購入。

3人の共通項: **「食」が旅の目的で、食情報の検索が旅マエ〜旅ナカに集中し、その全場面で通信が必要**。グルメ中心はTOFU偏重ではなく、購買文脈（旅マエ準備）に最初から乗っている。

## 3. コンテンツアーキテクチャ：Hub & Spoke

### 3-1. 二層構造

```
【グルメクラスタ（Spoke）— 月産の6割・AI引用と集客の資産】
  福岡クラスタ（旗艦・最優先）
    屋台・ラーメン・もつ鍋・鮮魚・うどん・清川高砂ローカル・
    空港から市内・yah.homes周辺 …「在住者の今」を武器に
  → 東京クラスタ → 大阪クラスタ（Phase 2〜)
        │
        │ 通信ブリッジ（全記事に設置）
        ▼
【マネーページ（Hub）— 8本×4言語だけを最高品質で維持】
  1. 日本eSIM完全ガイド（購入〜開通）
  2. iPhone設定 / 3. Android設定（HowTo Schema）
  4. Airalo比較（llms.txtの価格データと連動）
  5. 日数×用途別データ容量シミュレーション
  6. 繋がらないときの対処法
  7. トップアップ方法
  8. FAQ 30選（FAQPage Schema）
  → yah.mobi 購入フロー
```

v5の56本リストは資産として維持するが、カテゴリA(20本)は上記8本に統合・圧縮する。**マネーページは「数」ではなく「鮮度と網羅性」で勝つ**（公開後も毎月更新。iOS/Android の新バージョン対応、価格改定の即日反映）。

### 3-2. 月産10本の配分（編集者1名体制）

| 種別 | 本数/月 | 役割 |
|---|---|---|
| グルメ一次情報（福岡→東京・大阪） | 6本 | 集客・AI引用・yah.homes連携 |
| マネーページ新規 or 大規模更新 | 2本 | 回収装置の維持（8本完成後は更新に充当） |
| 旅行×通信（季節性：桜・花火・紅葉・年末） | 2本 | 季節検索の刈り取り |

### 3-3. グルメ記事の品質基準（AI引用の生命線）

一次情報チェックリスト — 全項目を満たさない記事は公開しない：
- [ ] 実訪問している（訪問日を記事に明記）
- [ ] 実価格・実メニュー（撮影した価格表・伝票）
- [ ] オリジナル写真5枚以上（ストック写真禁止）
- [ ] 地図・アクセス・営業時間・行列傾向（曜日/時間帯）
- [ ] 「スマホがその場で必要になる場面」の具体記述（→通信ブリッジへ接続）
- [ ] 直接回答ブロック・FAQ 3問以上・Schema Markup（v5 §5の7ブロックテンプレートを継承）

## 4. 転換設計（v5 §3を継承・実装条件を追記）

- 7ブロックテンプレート・CTA配置・通信ブリッジ・yah.homes クロスCTA は **v5の設計をそのまま採用**（§5・§7は有効）。
- 追加①: 全CTAリンクは `?ref=` / UTM を必須とし、**M3実装（affiliateCode 計測基盤）とGA4で記事→購入の実CVRを記事単位で計測**する。CVR実測が出た時点で、財務モデルのCVR入力（現状1%）を実測値に置換する。
- 追加②: グルメ記事中心のミックスはCVRが1%を下回る可能性がある（TOFU比率上昇）。**感度: CVR 0.6%なら magazine 単体の42ヶ月損益は約±0**。ゆえにマネーページ8本の品質と通信ブリッジの文面テストが本戦略の生命線であり、月次でブリッジ文面のA/Bを回す。

## 6. 制作体制と週次リズム（編集者確保済み前提）

| 役割 | 担当 | 内容 |
|---|---|---|
| 戦略・最終品質判断 | Yoshi | 週次30分レビュー。福岡実訪問の同行/撮影（月2回） |
| 制作統括 | 編集者（外注・確保済み） | 企画・執筆/AI下書き編集・入稿・SNS転載・鮮度管理 |
| 下書き・ローカライズ | AI（Manus/Claude） | 日本語ファースト→KO/繁中/EN 同時展開（v5 §9の自動化フロー継承） |
| ネイティブチェック | 韓国語・繁体字チェッカー（採用: 1ヶ月以内） | 月次まとめチェック |

- 週次リズム: 月曜=企画確定（2〜3本）→ 水木=執筆・ローカライズ → 金=Yoshiレビュー・公開・IG転載。
- 予算: 編集者＋ライター/チェッカー＋撮影実費 ≒ **月¥18万**（財務モデルの入力値と一致。超える場合はシート B47 を更新）。
- 鮮度管理（v5 §8-3継承): 公開1週間後に順位確認、1ヶ月後にAI引用確認、3ヶ月ごとに価格・営業情報を更新。**グルメ記事の「更新日」はAI引用の主要シグナル**。

流通（Phase 1 は絞る）: 各記事につき Instagram 1投稿（3言語画像テンプレ化）のみ。TikTok・Naver Blog・Dcard・LINE OA は Phase 2 で市場展開に合わせて追加（空白地帯レポートの優先度準拠）。メールマガジンは Phase 2（記事30本・月間訪問3,000到達後）に導入。

## 8. KPI（財務モデル接続版）

Base = 42ヶ月シート magazineシナリオの出力（M1開始に前倒し・要シート更新）。Stretch = 訪問/記事が250に上振れした場合。

| KPI | M3 | M6 | M12 | M24 |
|---|---|---|---|---|
| 公開記事数（日本語原稿） | 30 | 60 | 120 | 240 |
| 成熟記事数（公開6ヶ月経過） | 0 | 0 | 60 | 180 |
| 月間オーガニック訪問（Base） | 500 | 1,500 | 9,000 | 27,000 |
| 月間販売（Base / CVR1%） | — | 15 | 90 | 270 |
| 月次損益（Base） | −¥18万 | −¥14万 | −¥1万前後 | **+¥36万** |
| AI引用（補助指標※） | 計測開始 | 月20件 | 月100件 | 月300件 |

※ AI引用の計測方法: (a) GA4 の参照元 chatgpt.com / perplexity.ai / gemini、(b) Search Console の AI Overviews 表示クエリ、(c) 月次で主要20クエリを ChatGPT/Perplexity に投げて引用有無を目視記録（編集者の月次タスク）。

**単体の損益より重要な位置づけ**: 統合シナリオ（鯨＋magazine）では magazine の先行赤字を鯨の早期黒字が吸収し、**事業全体の累計黒字化は M13**。magazine は「月100万円営業利益」の後半（M24以降）を支える複利エンジンである。

## 9. リスクと対応

| リスク | 対応 |
|---|---|
| グルメ中心でCVRが1%を切る | マネーページ8本＋ブリッジ文面の月次A/B。CVR実測をシートに反映し早期に判断（感度: 0.6%で±0） |
| AI検索のさらなるゼロクリック化 | 一次情報比率を上げる（AIが生成できない情報だけ作る）。llms.txt・Schema・引用計測で「引用され率」を追う |
| 編集者の離脱 | SOP化（品質チェックリスト・テンプレート・週次リズム）を M1〜2 で文書化し属人化を防ぐ |
| 訪問単価の下振れ（150→100未満） | 福岡クラスタのAI引用実績をM3末に確認し、未達なら記事タイプ配分を見直し（比較・季節系へシフト） |
