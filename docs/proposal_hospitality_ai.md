# 提案書 — ホスピタリティガイドの AI 統合と管理

対象: chat-yah-mobi（AI チャットサポート）
入力資料: `hospitality-guide.md`（リッツ・カールトン／フォーシーズンズ／ディズニー HEARD／OONAS 等を統合した接客基準）
状態: **提案（実装前・承認待ち）**

---

## 1. 背景・目的

- 世界最高峰のホスピタリティ哲学を統合した資料を、AI チャットの応答品質（トーン・共感・WOW・感情マネジメント）向上に活かしたい。
- 併せて、この基準を**管理画面から運用（追加/編集/ON-OFF）**できるようにしたい。

## 2. 重要な前提：「学習」＝モデル訓練ではなく in-context guidance

| 方式 | 内容 | 採否 |
|---|---|---|
| **A. プロンプト注入（in-context）** | ガイドを“行動指針”としてシステムプロンプトに入れ、応答のたびに適用 | ✅ **採用** |
| B. Fine-tuning（モデル再訓練） | Gemini を専用データで再訓練 | ✗ 高コスト・更新が遅い・不透明・Vertex では別運用 |

- **A の利点**: 管理画面から**即時更新**／訓練コストゼロ／どの指針が効いているか透明／言語・状況で出し分け可能。
- **現状すでに A の土台がある**: `chat_hospitality_guidelines`（`isActive`/`priority`/`title`/`content`）→ `functions/src/utils/ai.ts` の `loadHospitalityGuidelines()` が active な基準を優先度順に読み、`generateAIResponse` のシステムプロンプトへ注入。
- → 提案は「**中身を充実**」「**管理UIを用意**」「**状況別に出し分け**」の3点。

## 3. 統合設計：ガイドを「どこに」入れるか

ホスピタリティは「**どう振る舞うか（behavior/tone）**」であり、RAG（`chat_rag_documents`）が担う「**何を答えるか（事実知識）**」とは別レイヤ。

| ガイドの内容 | 種別 | 注入方法 |
|---|---|---|
| クレド／モットー／ゴールデンルール（第1章） | 常時 | システムプロンプト（常時・最優先） |
| 3ステップ・オブ・サービス（第2章） | 常時 | システムプロンプト（常時） |
| トーン原則・禁止/推奨表現（第6章） | 常時 | システムプロンプト（常時・高効果） |
| 感情レベル別対応・HEARD・デエスカレーション（第4章） | 常時（コンパクト） | システムプロンプト（常時） |
| OONAS 5原則（第9章） | 常時 | システムプロンプト（常時・要約版） |
| 返金/技術未解決/怒り への対応プロトコル（第7章） | **状況別** | 検出時のみ注入（トリガー：感情レベル/キーワード） |
| WOW体験・権限委譲（第8・9章 Nordstrom） | 常時（要約） | システムプロンプト（軽く） |
| KPI・継続改善・OONAS採点（第10章・9.7） | **運用/評価用** | プロンプトには入れない（品質測定に活用＝第7項） |

**設計原則**: ガイド全文をそのまま毎回入れると**トークン費用・遅延・指示の希釈**が起きる。→ **原子的な指針（1〜3文）に分解**し、`priority` で「常時コア（高優先・~15〜20件）」を組み立て、状況別プロトコルはトリガー時のみ足す。

## 4. データモデル（`chat_hospitality_guidelines` の拡張）

現行フィールドに数個追加するだけ（後方互換）:

| フィールド | 型 | 用途 |
|---|---|---|
| `title` / `content` | string | 指針の見出し／本文（1〜3文推奨） |
| `category` | string | `credo`/`steps`/`tone`/`emotion`/`oonas`/`scenario`/`wow` 等（グルーピング） |
| `priority` | int | 小さいほど先・常時コアの選抜順 |
| `isActive` | bool | ON/OFF |
| `scope` | `"always"` \| `"situational"` | 常時注入 or トリガー時のみ |
| `trigger`（scope=situational時） | map | 例: `{ emotionLevel: [3,4] }` や `{ keywords: ["返金","refund"] }` |
| `lang`（任意） | string\|null | 言語固有の例文を分ける場合（原則は言語非依存の指針を推奨） |

## 5. AI 側の組み込み（`ai.ts`・要コード変更＝別途承認）

- `loadHospitalityGuidelines()` を拡張:
  1. `scope=="always"` かつ `isActive` を `category`＋`priority` 順に整形（カテゴリ見出し付き）。
  2. トークン上限を意識し、**常時コアは件数/文字数で上限**（例: 上位N件 or 合計X字）。
- 状況別注入（第2段階）: `onVisitorMessageCreated` で感情レベル/キーワードを軽く判定し、該当する `situational` 指針だけを追加注入。
- 既存の `generateAIResponse` のシステムプロンプト組み立てに、整形済みホスピタリティブロックを差し込む（現状の `params.hospitalityPrompt` 経路を拡張）。

## 6. 管理画面（admin）

> 注: ミニマル化で一度 `Hospitality` 画面を削除済み。本提案で**再追加**する（要件が明確になったため・意図的な方針転換）。

- **`/admin/hospitality`（再追加）** で `chat_hospitality_guidelines` を CRUD:
  - 一覧（category でグループ表示・priority 並べ替え・isActive トグル・scope 表示）。
  - 追加/編集フォーム（title/content/category/priority/scope/trigger）。
  - **プロンプトプレビュー**（現在 active な常時コアが実際どう組み上がるかを表示）＝運用者が“効き方”を確認できる（任意・推奨）。
- RAG 管理（`/admin/rag`）とは**別メニュー**にする（知識 vs 振る舞いの分離を維持）。
- 既存の `chat_hospitality_guidelines`（移行済み2件）はそのまま活かし、ガイドを分解した指針を追加投入。

## 7. 品質測定（任意・発展）

ガイドの KPI／OONAS 採点（9.7）を評価ループに活用:
- **LLM-as-judge**: 終了セッションを OONAS 5軸（各20点）でスコアリングし、低スコアパターンを可視化 → 指針を改善（第10章 PDCA の自動化）。
- CSAT（アンケート `chat_surveys`）と突き合わせて効果測定。
- ※ これは発展課題（まずは 5・6 で品質を上げてから）。

## 9. リスク・考慮

- **トークン費用/遅延**: システムプロンプト肥大 → 常時コアは上限管理・状況別は必要時のみ。
- **過剰制約**: 指針が多すぎると応答が硬直化 → 「原則＋良い判断（Nordstrom）」のバランスを保つ。
- **多言語**: ガイドは日本語中心だが**原則は言語非依存**。例文は言語固有になりうるので、指針は抽象度を上げ、応答言語は既存の言語検出に委ねる。
- **admin 再追加**: ミニマル方針の一部を戻すが、要件が明確なので妥当。
- **Functions 変更**: `ai.ts` 改修はガードレール上ユーザー承認が必要（本提案で承認を求める）。
