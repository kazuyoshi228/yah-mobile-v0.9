# 🚀 yah.mobile: BaaS-First アーキテクチャ移行 完了レポート

「MySQLのようなレガシーなDB思想」を完全に払拭し、**「Firebaseの強みを最大限に引き出す（BaaS-First）」** というコンセプトのもと、システム全体の構造と運用フローの改修がすべて完了しました。

本レポートでは、実装された具体的な改善内容と、システムにもたらす圧倒的なメリットについて総括します。

---

## 1. データ構造の「非正規化」によるN+1問題の完全解消
RDB（リレーショナルデータベース）における「JOIN」の思想を捨て、必要なデータは保存時に1箇所にまとめる（非正規化）設計を導入しました。

- **`esim_activations` / `esim_links` へのメタデータ埋め込み**
  - **変更前:** アプリで「eSIM一覧」を表示する際、プラン情報を取得するために毎回別コレクションへアクセス（N+1クエリ）が発生していました。
  - **変更後:** 注文時に `planName`（例: 日本 10GBプラン）や `totalDataGb` を発行ドキュメント内に直接書き込むようにしました。これにより、アプリは **「1回のクエリ」で画面描画に必要なすべての情報を爆速で取得** できるようになりました。
- **`orders` への顧客属性の埋め込み**
  - `userEmail` や `userName` を注文ドキュメントに直接記録し、管理画面で顧客情報を別途引き当てる手間をゼロにしました。

## 2. リアルタイム・ダッシュボード集計の自動化
管理画面（`/admin`）で売上や注文件数を表示するために、毎回数万件のデータを集計する重い処理（MySQLの `COUNT()` や `SUM()` に相当する処理）を排除しました。

- **`system_stats` コレクションの新設**
  - Stripeの決済完了時（Webhook）に、Firebaseの強力な機能である `FieldValue.increment()` を活用し、専用のドキュメント（`stats/global`）の「注文件数」と「売上合計金額」をアトミックに加算する処理を実装しました。
  - これにより、管理画面を開いた瞬間に **「常に最新の売上データ」が一瞬で表示** される極めて効率的なアーキテクチャとなりました。

## 3. 自動クリーンアップ（TTL）によるランニングコスト最適化
- **Stripe Webhook と リトライジョブの `expiresAt` 追加**
  - ログやエラー追跡のために保存している `stripe_events`（決済重複防止用）や `esim_retry_jobs`（エラー時の自動再試行用）に対して、一定期間で不要になるデータの寿命（Time-to-Live）として `expiresAt` フィールドを追加しました。
  - 今後、Firebaseコンソール上でTTLポリシーをONにするだけで、古いデータがGoogle側で自動的に削除され、将来のDB維持コストを極小化できます。

## 4. サポート体制のハイブリッド化（文脈の自動連携）
カスタマーサポートの質とスピードを劇的に向上させるため、問い合わせ時のデータ連携を強化しました。

- **`ContactSection.tsx` の改修**
  - ログイン済みユーザーがフォームを送信する際、フロントエンド側で自動的に直近の注文（`orders` の最新1件）を検索し、`userId` と一緒に `orderId` を送信データ（`contact_inquiries`）に埋め込みます。
  - これにより、サポートオペレーターやAIチャットボットは、「誰の・どの注文に対するトラブルか」を事前に把握した状態で対応を開始できます。

## 5. 論理削除（Soft Delete）の徹底によるデータ保護
- **管理画面における `deleteDoc` の全廃止**
  - 以前は管理画面からプランや比較表の行を削除すると、物理的にデータが消え、過去の注文履歴から商品情報が参照できなくなるリスクがありました。
  - 今回の改修で、これらをすべて `updateDoc(docRef, { isActive: false })` の論理削除に切り替えました。これにより、イミュータブル（不変）な売上履歴を恒久的に保護するエンタープライズ水準の安全性を確保しました。

> [!TIP]
> ### 運用担当者様へのご案内（次のアクション）
> プログラム側の実装はすべて完了していますが、GCP（Google Cloud）側で1点だけ設定をお願いします。
> 1. **Firebase Console** の Firestore Database > 「インデックス（Indexes）」 > 「TTL」タブを開く。
> 2. コレクショングループ `stripe_events` と `esim_retry_jobs` に対して、タイムスタンプフィールド `expiresAt` を対象としたTTLポリシーを作成してください。

---

### まとめ
今回の改修により、yah.mobile のバックエンドは単なる「データ置き場」から、**「高速表示」「自動集計」「文脈共有」「コスト最適化」を自律的に行う強力なエンジン**へと進化しました。このBaaSを極限まで活用するアーキテクチャにより、今後のビジネスの急拡大にもサーバーダウンの心配なく対応可能です！
