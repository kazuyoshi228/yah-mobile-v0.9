# システム障害および返金対応パターン定義書

本ドキュメントは、yah.mobi（v4）の「Firebase + Stripe + Bappy」構成において、「お客様の決済は完了したものの、正常にサービスを提供できない状態（＝弊社側の過失）」に陥るパターンと、その検知・対応フローをまとめたものです。

> [!NOTE]
> 基本的な利用規約として「お客様都合の返金は不可」としている前提のもと、**システムエラーに起因する正当な返金対象**を明確に定義し、運用コストを最小化することを目的とします。

---

## 1. 決済と発行のタイムラグで起きるエラー（パターンA）

お客様がStripeで決済を完了した直後に、システムの連携が途切れてしまい「お金だけ引き落とされた状態」になるケースです。**発生確率が最も高く、自動検知が可能な領域**です。

### 1-1. Bappy（eSIMベンダー）のAPIダウン・在庫切れ
- **原因**: Stripeでの決済成功通知を受けてシステムがBappyに「eSIMを発行」とリクエストした際、Bappy側がシステムメンテナンス中だったり、APIがタイムアウトしたりするケース。
- **データ状態**: Stripeの決済＝完了 / DBのオーダーステータス＝`paid` / **QRコード（`esim_links`）＝存在しない**

### 1-2. Stripeからの「決済完了通知（Webhook）」の受信失敗
- **原因**: 決済直後、StripeからFirebase（Cloud Functions）へ「支払い完了」という通知が飛ぶ際、ネットワークエラーやFirebase側の瞬間的な障害で通知を取りこぼすケース。
- **データ状態**: Stripeの決済＝完了 / **DBのオーダーステータス＝`pending`のまま** / QRコード＝存在しない

### 1-3. データベース（Firestore）の保存エラー
- **原因**: Bappyから正常にQRコードのデータを受け取ったが、それをデータベース（`esim_links`）に書き込む瞬間にFirebaseがエラーを起こし保存に失敗するケース。（※Bappy側には弊社からの発注費用が発生してしまっている状態）
- **データ状態**: Stripeの決済＝完了 / DBのオーダーステータス＝`paid` / **QRコード＝存在しない**

---

## 2. 発行後の不良品・通信障害（パターンB）

QRコード自体はお客様の画面に表示されているものの、サービスとして成立していないパターンです。こちらはデータベース上のステータスだけでは判定が難しいため、お客様からの申告ベースでの調査が必要になります。

### 2-1. QRコード（LPAプロファイル）の生成不良
- **原因**: Bappyから送られてきたQRコードのデータ自体が破損している、またはApple/Android側の仕様変更で読み込めない無効な形式になっているケース。
- **データ状態**: お客様がQRコードをスキャンしても「このコードは無効です」とスマホに表示される。

### 2-2. 日本国内での通信障害（ローミング先の障害）
- **原因**: eSIMのインストールまでは成功したが、提携している国内キャリア（docomo/au/SoftBank等）やBappyの基幹回線で大規模な通信障害が起きているケース。
- **データ状態**: スマホのアンテナは立つ（あるいは圏外）が、一切インターネットに繋がらない。

### 2-3. フロントエンド（ブラウザ）のクラッシュ
- **原因**: データベース上には正常にQRコードが保存されているが、お客様のスマホのブラウザ環境などの理由で、サイト上でQRコードの画像を描画する処理に失敗して画面が真っ白になるケース。
- **データ状態**: システム上は完璧に発行されているが、お客様の目にはQRコードが見えない。

## 3. 運用上の対策と自動化フローの提案

> [!TIP]
> 上記の「パターンA」はすべて**『Stripeで決済が成功しているのに、DB上にQRコード（esim_links）が存在しない』**という共通のデータ状態になります。これを活用することで、カスタマーサポートを無人化（AI化）することが可能です。

### 推奨されるサポートフロー（AIチャットボット活用）

将来的にAIチャット（または夜間バッチ処理）に以下のロジックを組み込むことで、保守運用の外注なしに24時間対応が可能となります。

1. **ユーザーからの問い合わせ**
   - 「購入したのにQRコードが出ません」という問い合わせが入る。
2. **AIによるステータス自動確認**
   - AIが裏側のツールを使い、対象ユーザーの注文状態をチェックする。
   - `Stripe決済状態 == 成功` かつ `Firestore esim_links == 存在しない` ことを確認する。
3. **AIによる自動対応（分岐）**
   - 上記の条件に合致する場合：弊社側のシステムエラー（パターンA）と断定。
   - お客様に「エラーにより発行に失敗しました。全額返金するか、再度発行を試みるかお選びください」と案内する。
4. **自動返金・再発行の実行**
   - お客様が「返金」を選べば、AIが直接Stripeの返金APIを叩いて処理を完了させる。

> [!IMPORTANT]
> **悪用防止のガードレール**
> AIが騙されて無条件に返金してしまうことを防ぐため、返金を実行するAPI（Cloud Functions）側には、必ず「通信量が0バイトであること」や「システムエラーの条件を満たしていること」を強制チェックするロジックを実装します。
